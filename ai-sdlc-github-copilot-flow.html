2 Key differences from Claude Code

Architectural differences
OrchestrationClaude Code uses a single session with Task tool for subagent forking. Copilot uses separate GitHub Actions jobs — contract first, then BE + FE as parallel jobs, then a merge job. More YAML, same outcome, fully deterministic.
Agent delegationCopilot supports intent-based auto-delegation in IDE: the runtime reads agent descriptions and matches user intent. In CI, we use explicit --agent <name> for determinism.
Hook format.github/hooks/hooks.json (JSON config) vs .claude/settings.json. Copilot hooks deny via {"permissionDecision":"deny"} on stdout, not exit code 2. A normalize shim bridges both formats so hook scripts are shared.
InstructionsCopilot adds applyTo: glob scoping — instructions auto-activate per file type. Claude loads everything unconditionally.
SkillsIdentical. Same open standard SKILL.md files. Both runtimes read .github/skills/ and .claude/skills/.
Project layerShared. Both read from .claude/project/ — no duplication.

4 Implementer setup — GitHub Copilot

Prerequisites

You need: A GitHub repo, Copilot CLI installed (npm i -g @github/copilot-cli), an ADO project with a PAT, a Slack incoming webhook URL, and repository admin access.
1
Copy the starter into your repo
copy ai-sdlc-orchestrator <your-repo>
This puts .claude/ (shared layer), .github/ (Copilot entrypoint), and docs/ in place.
2
Run bootstrap (via Copilot or Claude Code — either works)
copilot --prompt "/bootstrap-project"
Or: claude -p "/bootstrap-project"
Both read the same SKILL.md. Output: .claude/project/PROFILE.draft.md.
3
Review and activate the profile
Fix [NEEDS HUMAN INPUT] entries. Rename to PROFILE.md.
4
Fill in guideline stubs
Edit .claude/project/guidelines/*.md and .claude/project/overrides/. Both Copilot and Claude Code read these.
5
Create GitHub repository secrets
Settings → Secrets and variables → Actions:
COPILOT_TOKEN, ADO_PAT, SLACK_WEBHOOK_URL.
Also set repository variables: ADO_ORG_URL, ADO_PROJECT.
6
Enable the workflows
The four YAMLs in .github/workflows/ activate on push. Check Actions tab to confirm all four appear. design-gen also supports workflow_dispatch for manual testing.
7
Wire the ADO → GitHub dispatch
ADO Project Settings → Service Hooks → Generic webhook → target your GitHub repo's repository_dispatch endpoint. Event type: ado-workitem-ai-ready. Map work_item_id in the payload.
Alternatively: use a small Azure Function or Logic App to bridge ADO webhooks to GitHub's /dispatches API.
8
Set branch protection rules
Settings → Rules → Rulesets: require 1 human reviewer on main and design/**, require status checks to pass, prevent direct pushes.
9
Commit and push
git add .claude/ .github/ docs/ && git commit -m "Adopt AI-SDLC" && git push
10
Test manually first
Actions → design-gen → Run workflow → enter a work item ID. Watch the run. If it produces a design PR, the wiring works. Then enable the ADO dispatch for automatic triggering.
Copilot CLI is newer than Claude Code. Expect some rough edges in headless mode — especially around --agent flag handling and auth in CI. Pin the CLI version in your workflows once you find one that works.
