# AI-SDLC — starter repo

A complete AI-assisted SDLC scaffold for Claude Code + Azure DevOps. Covers requirements → design → slicing → parallel implementation → three-subagent PR review. Human gates at every decision point.

## The design docs

Read these first. Everything else implements them.

- **[docs/FUNCTIONAL_DESIGN.md](docs/FUNCTIONAL_DESIGN.md)** — what this system does and why
- **[docs/TECHNICAL_DESIGN.md](docs/TECHNICAL_DESIGN.md)** — how it's built; specs for every file in this repo

## Onboarding a new repo (one-time, per repo)

```bash
# 1. Copy this starter into the target repo (or add as submodule under .claude-starter/)
cp -r ai-sdlc-starter/.claude <target-repo>/
cp -r ai-sdlc-starter/.azure-pipelines <target-repo>/

# 2. From inside the target repo, run bootstrap
cd <target-repo>
claude -p "/bootstrap-project"

# 3. Review the generated draft profile
$EDITOR .claude/project/PROFILE.draft.md
# Fix [NEEDS HUMAN INPUT] entries, confirm stack detection.

# 4. Rename to activate
mv .claude/project/PROFILE.draft.md .claude/project/PROFILE.md

# 5. Fill in the guideline stubs
$EDITOR .claude/project/guidelines/*.md
$EDITOR .claude/project/stacks/*.md       # whatever bootstrap generated

# 6. Commit
git add .claude/ .azure-pipelines/
git commit -m "Adopt AI-SDLC"
```

## Pipeline setup (one-time, per ADO project)

1. **Variable group `ai-sdlc-secrets`** (Project Settings → Library):
   - `ANTHROPIC_API_KEY` (secret)
   - `ADO_PAT` (secret, scoped: Work Items RW, Code RW, Build RW)
   - `SLACK_WEBHOOK_URL` (secret)

2. **Service connection `ado-service-hook`** for webhook ingress.

3. **Create four pipelines**, each pointing at its YAML:
   - `design-gen` → `.azure-pipelines/design-gen.yml`
   - `implement-story` → `.azure-pipelines/implement-story.yml`
   - `pr-review` → `.azure-pipelines/pr-review.yml`
   - `post-merge` → `.azure-pipelines/post-merge.yml`

4. **Service Hook** (Project Settings → Service Hooks):
   - Type: Work item updated
   - Filter: Tags contain `ai:ready`
   - Action: Trigger `design-gen` pipeline, map `workItemId` from payload

5. **Branch policies** on `main` and `design/**`:
   - Require 1 human reviewer
   - Require PR build to pass
   - Prevent direct pushes

6. **Custom work item tag**: Create `ai:ready` in the process template.

## Running a story — the normal workflow

1. PM creates a work item in ADO, writes a description, tags `ai:ready`.
2. `design-gen` pipeline fires within a minute. Generates `docs/designs/WI-<id>/` + child Task work items + a design PR.
3. Tech lead reviews the design PR. Merges when satisfied.
4. `implement-story` fires on merge. Orchestrator delegates to subagents. Opens a feature PR.
5. `pr-review` fires on the feature PR. Three subagents post review comments in parallel.
6. Human reviews and merges. `post-merge` closes the ADO work items.

## Critical rules (the ones worth repeating)

- **Master skills are read-only.** Customize via `.claude/project/`, never by editing `.claude/skills/`.
- **Hooks are the enforcement layer.** If you disable them, the whole trust model collapses.
- **Claude never merges.** Branch policies enforce this. Do not remove them.
- **Every run produces a trace.** Archived as pipeline artifact. Read it when things go wrong.
- **CLI over MCP.** If you add a new integration, write a script wrapper first. MCP is reserved for things with no stable CLI.

## What to expect in week one

- Design generation works well on clear requirements; less well on vague ones. The `[NEEDS HUMAN INPUT]` markers are a feature — they prevent silent fabrication.
- Slicing tends to be too fine on the first few runs. Expect to correct the slice YAML in the design PR before merging.
- Subagent parallelism works when file boundaries are clean. Merge conflicts at the orchestrator are a signal that the boundary is wrong — fix the scope-map, not the code.
- Review subagents produce the most immediate value even if nothing else is adopted. Start there if you're pressure-testing.

## Known gotchas

1. **`ClaudeCodeBaseTask@1` version lock.** Pin the task version in all four YAMLs; upstream changes have caused regressions.
2. **`az boards` extension version.** Pin it in pipelines (`az extension add --version`) once you find a working version in your tenant.
3. **`--dangerously-skip-permissions`** is used in pipelines because hooks are the real guardrail. Never use this flag locally.
4. **Work item fields** referenced by skills (System.Parent, System.Tags) use ADO default names. If your process template renames them, update `ado-wi-create-slice.sh` and `ado-wi-update.sh`.
5. **`claude -p` argument parsing** — some shells quote-mangle multi-word prompts. Pipeline YAML handles this; local testing can trip on it.

## What's not included

- Stack files (`.claude/project/stacks/*.md`) are generated by `/bootstrap-project` based on what it detects; no generic template ships with this starter.
- End-to-end test runners. The system checks unit + integration; E2E stays with your existing harness.
- Cost dashboards. Read the trace artifacts with `jq` or wire into your observability stack.
